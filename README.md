# ReplyWise AI

**A retrieval-grounded AI assistant that drafts suggested email replies — and, more importantly, measures how good they are.**

ReplyWise AI takes an incoming business email, retrieves the most similar past email/reply pairs, and uses a generative model to draft a suggested response grounded in those examples. It then scores every generated reply with a transparent, multi-metric evaluation system that explains *why* each reply is good or bad.

The project deliberately spends most of its weight on **evaluation**, because for a suggested-reply system the hard question is not "can it produce text?" but "can you trust the text, and can you prove it?"

## What it does

1. **Ingests** a labeled dataset of business email/reply pairs across 10 categories.
2. **Retrieves** the top-k most semantically similar past emails for a new incoming email.
3. **Generates** a suggested reply with a Claude (or OpenAI-compatible) model, grounded in the retrieved examples via few-shot prompting.
4. **Evaluates** the reply against ground-truth labels using eight complementary metrics plus an optional LLM-as-judge.
5. **Reports** per-reply scores with explanations, and an aggregate system report with category breakdowns, best/worst examples, and a failure analysis.

The whole system runs **with zero credentials** in a deterministic mock mode, so it is fully reproducible offline for reviewers and CI.

## Quickstart

```bash
# 1. Install (a virtualenv is recommended)
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 2. Generate the synthetic dataset (~220 labeled examples)
python -m src.dataset.generate_dataset --n 220 --seed 7

# 3. Build the retrieval index
python -m scripts.build_index

# 4. See a single suggested reply + its evaluation
python -m scripts.demo

# 5. Run the full evaluation harness over the held-out test set
python -m scripts.run_evaluation

# 6. (optional) Launch the demo UI or the API
streamlit run src/ui/app.py
uvicorn src.api.main:app --reload
```

To use a real model, copy `.env.example` to `.env` and set a provider key. The quickest option is **Groq** (OpenAI-compatible, fast, generous free tier):

```bash
REPLYWISE_PROVIDER=groq
GROQ_API_KEY=your_key_here          # from https://console.groq.com
REPLYWISE_MODEL=llama-3.3-70b-versatile
```

`ANTHROPIC_API_KEY` and `OPENAI_API_KEY` are also supported. With **no** key set, ReplyWise runs a **nearest-neighbor mock generator** so every command above still works offline.

## The dataset

A synthetic-but-realistic corpus of business email/reply pairs, generated by `src/dataset/generate_dataset.py`. Each record is a fully labeled `EmailExample`:

| Field | Purpose |
|---|---|
| `incoming_email` | the message to reply to |
| `gold_reply` | a reference "good" reply |
| `category` | one of 10 business categories |
| `intent` | the sender's underlying goal (e.g. `propose_meeting_time`) |
| `tone` | the register a good reply should adopt (e.g. `empathetic-professional`) |
| `required_facts` | facts a correct reply **must convey** (e.g. a corrected invoice total) |
| `must_include` | key phrases/entities that should appear |
| `must_not_include` | phrasing that must **not** appear (e.g. "calm down", "pay immediately") |
| `ideal_action` | the action a good reply drives toward |
| `risk_level` | `low` / `medium` / `high` — how costly a bad reply would be |

**Categories:** scheduling, recruiting, customer support, sales/demo requests, invoice/payment, internal project update, partnership, event logistics, complaint/escalation, follow-up.

Around **220 examples** (configurable, 150–300 recommended), balanced across categories, split ~85/15 into `data/synthetic_emails.jsonl` (retrieval corpus) and `data/test_emails.jsonl` (held-out evaluation set).

The generator composes each email from per-category templates with randomized slot fills (names, companies, dates, amounts, ticket/invoice numbers). Crucially, each **gold reply is constructed to contain its `required_facts` and `must_include` phrases and to avoid its `must_not_include` phrases** — so the evaluation labels are internally consistent by construction, and the metrics have a trustworthy ground truth to score against.

## Why synthetic data

Real inboxes are the obvious data source and the wrong one to publish:

- **Privacy.** Real business email contains names, contracts, prices, and personal data that cannot be shared publicly or used without consent.
- **Controlled labels.** The evaluation depends on knowing what a correct reply *must* contain and *must not* contain. Real data has no such labels; hand-labeling hundreds of emails is slow and subjective. Synthetic generation lets us attach precise, machine-checkable labels to every example.
- **Coverage and balance.** We can guarantee representation across all 10 categories and across risk levels, including rare-but-critical cases like complaint escalations.
- **Reproducibility.** A fixed seed regenerates the exact dataset, so results are comparable across runs and machines.

The trade-off — synthetic emails are cleaner and less adversarial than real ones — is discussed under [Limitations](#limitations). The architecture is designed to swap in a real, privately-held dataset without any code change: same schema, same pipeline.

## Generation approach (RAG)

ReplyWise uses **retrieval-augmented generation** rather than a fine-tuned or classical classifier, because the task is open-ended text generation that must stay grounded in prior examples and specific facts.

```
incoming email
      │
      ▼
[ embed ]  sentence-transformers/all-MiniLM-L6-v2
      │
      ▼
[ retrieve top-k ]  FAISS / NumPy cosine over the training corpus
      │
      ▼
[ build prompt ]  system guardrails + k few-shot email/reply exemplars
      │
      ▼
[ generate ]  Claude / OpenAI-compatible / mock
      │
      ▼
suggested reply + retrieved context + metadata
```

**Why RAG over the alternatives:**

- **vs. a classifier** — the challenge asks for generated replies, not a label. A classifier can route or template but can't compose a fact-specific, tone-appropriate response.
- **vs. zero-shot generation** — without retrieval the model has no house style and no grounding, so it hallucinates specifics and drifts in tone.
- **vs. fine-tuning** — fine-tuning is heavier, needs far more data, and is hard to update. RAG adapts instantly when you add examples to the corpus and keeps the reasoning inspectable (you can see exactly which examples grounded a reply).

The prompt (`src/generation/prompt_builder.py`) instructs the model to use **only facts present in the incoming email**, to **ask for clarification instead of inventing details**, and to match the situation's tone. The retrieved exemplars supply style and structure, not facts to copy.

**Mock mode.** When no API key is present, the generator returns the single most similar past reply — a legitimate, reproducible *nearest-neighbor baseline*. This keeps everything runnable offline and gives you a baseline that a real model should beat.

## Evaluation philosophy

Evaluation is the core of this project. A suggested-reply system fails in ways that a single similarity number cannot see: it can sound perfect while **omitting the one fact that mattered**, or **inventing a price that was never quoted**, or **being curt with an angry customer**. So ReplyWise scores each reply along several independent axes, each in `[0, 1]`, and combines them into a weighted final score with an explicit penalty for forbidden content.

Every axis is either **reference-based** (compared to the gold reply) or **reference-free** (judged against the labels or the input), and every score is **explainable** — the report states, in words, why each reply landed where it did.

---

## Metrics, formulas and weights

Implemented in `src/evaluation/metrics.py` and combined in `src/evaluation/evaluator.py`.

| Metric | What it measures | How it's computed |
|---|---|---|
| **Semantic similarity** | Does the reply mean the same as the gold? | Cosine similarity of sentence embeddings (reply vs. gold). |
| **Required-fact coverage** | Are the facts that *must* be conveyed present? | Fraction of `required_facts` found, allowing paraphrase (exact phrase match, or ≥70% significant-token overlap). |
| **Must-include coverage** | Are key entities/phrases present? | Same matcher over `must_include`. |
| **Must-not-include violation** | Did forbidden phrasing leak in? | Fraction of `must_not_include` present. `0` is clean; drives a penalty. |
| **Intent alignment** | Does the reply pursue the right goal? | Embedding similarity of the reply to a description built from `intent` + `ideal_action`. |
| **Tone match** | Is the register appropriate? | Lexicon-based trait matching against the expected tone, a politeness backbone bonus, and a hard penalty for rude markers. |
| **Helpfulness / completeness** | Is it structured and actionable? | Greeting + sign-off + a concrete next step + sensible length + fact coverage. |
| **Safety / no hallucination** | Are concrete specifics grounded? | Extract high-risk specifics (amounts, ticket/invoice numbers, times, %), check each appears in the incoming email, required facts, or retrieved context. |
| **LLM-as-judge** *(optional)* | Holistic second opinion | A model returns strict JSON scores + reasoning. Off by default; degrades to `None` without a key. |

**Final score (weighted):**

```
final_score =
      0.20 * semantic_similarity
    + 0.25 * required_fact_coverage
    + 0.15 * intent_alignment
    + 0.10 * tone_match
    + 0.15 * helpfulness
    + 0.15 * safety_no_hallucination
    - 0.50 * must_not_include_violation      # penalty, clipped to [0, 1]
```

Weights are defined in `src/config.py` and validated to sum to 1.0 at startup. **Required-fact coverage carries the most weight (0.25)** because conveying the right facts is the point of a reply; **safety and helpfulness are weighted equally with semantic similarity** so a fluent-but-wrong reply cannot score well.

**Pass/fail:** a reply passes if `final_score ≥ 0.70` **and** it has zero must-not-include violations. The violation is a hard gate, not just a deduction — a reply that tells an escalating customer to "calm down" should never pass, no matter how well it scores elsewhere.

## Why exact match is not enough

Exact (or near-exact) string match is the wrong tool for this task:

- **It punishes valid paraphrase.** "I'll send a calendar invite" and "I'll get an invite over to you" are equally correct; exact match scores one of them zero.
- **It rewards nothing about safety.** A reply can match the gold's wording and still be wrong if the situation's facts differ. Exact match has no concept of a hallucinated price or a missing fact.
- **It ignores tone and intent.** Two replies with identical token overlap can differ completely in whether they de-escalate an angry customer.
- **There is no single correct reply.** Many good replies exist for any email. A metric that assumes one gold answer is measuring the wrong thing.

ReplyWise instead measures *substance* (facts, intent, safety) and *manner* (tone, helpfulness), using the gold reply as one signal among several rather than the sole ground truth.

## Running the evaluation

```bash
python -m scripts.run_evaluation            # full held-out test set
python -m scripts.run_evaluation --limit 20 # quick subset
```

For each test email the harness retrieves from the training index (**excluding the item itself** to avoid leakage), generates a reply, and scores it. It writes:

- `outputs/evaluation_report.json` — full machine-readable results (every reply, every metric).
- `outputs/evaluation_report.md` — a human-readable report with category tables, best/worst examples, and failure analysis.

The aggregate report includes: **average score**, **pass rate**, **per-category scores**, **best and worst examples with explanations**, and a **common-failure-type tally** (missing facts, hallucination, tone mismatch, intent drift, must-not-include violations, ...).

Aggregate summary — **Groq `llama-3.3-70b-versatile` + real `all-MiniLM-L6-v2` embeddings**, 33 held-out test emails:

```
Average score : 0.834
Pass rate     : 81.8% (threshold 0.70)

Per-category:
  invoice_payment        avg=0.988 pass=100%
  partnership            avg=0.970 pass=100%
  project_update         avg=0.902 pass=100%
  recruiting             avg=0.859 pass=100%
  customer_support       avg=0.849 pass=100%
  ...
  event_logistics        avg=0.670 pass=50%
  scheduling             avg=0.683 pass=0%

Common failure types:
  missing_required_facts       6
  possible_hallucination       4
```

**Reading the failure analysis (this is the point of the system).** The model is strong on fact-heavy categories where the required facts live in the incoming email — invoices, partnerships, project updates all pass 100%. It struggles most on **scheduling**, and the reason is illuminating: a good scheduling reply *proposes a new time*, but the gold reply's specific time (e.g. "Monday at 10:30 AM ET") is labeled a `required_fact`, and any time the model invents is flagged by the hallucination check as an ungrounded specific. So the low scheduling score is partly a **real model behavior** (it can't know our calendar) and partly a **known metric tension** for propose-a-value tasks. Surfacing exactly this kind of insight — rather than hiding it behind a single number — is what the evaluation system is for. See [Limitations](#limitations) for how to fix it (per-category grounding rules).

> For a zero-setup, fully deterministic run (no API key, hash-fallback embeddings) the same harness produces a **nearest-neighbor mock baseline** at ~0.79 average — a floor for any real model to beat. Swap `sentence-transformers` and a real provider back in to reproduce the numbers above.

## Validating the metric itself

A scoring system is only trustworthy if the score correlates with human judgment. The plan to validate ReplyWise's metric:

1. **Build a small human-labeled validation set.** Sample ~50 generated replies spanning the score range and all categories.
2. **Collect human rankings.** Have 2–3 annotators independently rank or rate each reply for usefulness, correctness, and tone; measure inter-annotator agreement.
3. **Correlate metric vs. humans.** Compute rank correlation (Spearman/Kendall) between `final_score` and the human rating. A good metric shows strong positive correlation.
4. **Sanity-check the tails.** Confirm that high-scoring replies are genuinely useful and that low-scoring replies have real, identifiable problems (missing facts, wrong tone, hallucination).
5. **Tune the weights against human preference.** If, say, humans penalize hallucination more harshly than the metric does, raise the safety weight and re-check correlation. Use the failure analysis to find systematic gaps.
6. **Guard against gaming.** Verify the metric can't be fooled by keyword-stuffing (coverage uses token-overlap thresholds, not raw counts) or by verbosity (helpfulness penalizes over-length).

This turns the metric from an assertion into something falsifiable and improvable.

## Trade-offs

- **Synthetic vs. real data.** Synthetic data buys privacy, labels, and reproducibility at the cost of realism and adversarial variety. The schema and pipeline are built to drop in real data unchanged.
- **RAG vs. fine-tuning.** RAG is cheaper, updatable, and inspectable, but depends on retrieval quality and a good prompt rather than baked-in style.
- **Deterministic metrics vs. LLM-as-judge.** The core metrics are transparent, fast, and free but rely on hand-designed rules; the LLM judge is more holistic but slower, costlier, and non-deterministic. ReplyWise uses the deterministic metrics as the backbone and treats the judge as an optional cross-check.
- **Graceful fallbacks vs. accuracy.** The hash-embedding and mock-LLM fallbacks guarantee the project runs anywhere with no setup, but they are clearly weaker than the real components and are labeled as such at runtime.

---

## Limitations

- **Synthetic emails are cleaner than real ones** — shorter, less ambiguous, fewer multi-topic threads and edge cases.
- **Fallback modes are weak by design.** The hash embedder and nearest-neighbor mock exist for reproducibility, not quality; the headline numbers should be regenerated with real embeddings and a real model.
- **Tone scoring is lexicon-based** and can miss subtle or sarcastic tone; the optional LLM judge partially covers this.
- **The hallucination check targets structured specifics** (amounts, IDs, times, percentages) — the highest-risk category — but does not catch every kind of fabricated claim.
- **The metric has not yet been human-validated** on this repo; the [validation plan](#validating-the-metric-itself) describes exactly how to do it.
- **Propose-a-value tasks (scheduling) are penalized by strict grounding.** When a good reply must invent a plausible new value (a meeting time), the required-fact and hallucination checks — designed to catch fabricated invoice totals and ticket numbers — misfire. The fix is per-category grounding rules: for scheduling, credit *any* well-formed proposed time and only require that the reply offers one, rather than matching the gold's specific slot. This is a concrete, scoped next step, and the failure analysis already points right at it.

## Future improvements

- Human-validate and re-tune metric weights against annotator preference.
- Add a NLI-based factuality check for unstructured (non-numeric) claims.
- Report retrieval quality directly (precision@k, category-match rate).
- Add hard/adversarial cases: multi-intent emails, missing information, conflicting facts, hostile senders.
- Calibrate the pass threshold per risk level (stricter gates for high-risk categories like invoices and escalations).
- Stream generation and add a "regenerate with feedback" loop in the UI.

## How AI tools were used

This project was built with AI assistance, used deliberately and reviewed at every step:

- **Architecture and scaffolding.** An AI coding assistant helped lay out the module boundaries, the pydantic schemas, and the RAG + evaluation flow described above. Every design decision — the metric set, the weighting, the hard must-not-include gate, the fallback strategy — was chosen and reviewed intentionally, not accepted blindly.
- **Dataset generation.** The synthetic corpus is produced by a hand-designed template generator (not free-form model output), so the labels are controlled and consistent. AI assistance helped draft the per-category templates and realistic phrasing.
- **Generation runtime.** At runtime, a generative model produces the suggested replies. The system is model-agnostic — it supports **Groq** (used for the headline results above, via `llama-3.3-70b-versatile`), Anthropic Claude, and any OpenAI-compatible endpoint — and ships with a deterministic mock so it never *requires* a paid model to run or be evaluated.
- **Evaluation.** The metrics are deliberately mostly deterministic and human-readable so the scoring does not itself depend on an opaque model. An LLM-as-judge is offered as an *optional* cross-check, never as the sole arbiter.
- **Testing.** The test suite (pytest) pins the intended behavior of the metrics and the pipeline and runs fully offline.

The guiding principle: use AI to move fast on scaffolding and drafting, but keep the parts that determine *trust* — the evaluation logic and the dataset labels — transparent, deterministic, and human-owned.
